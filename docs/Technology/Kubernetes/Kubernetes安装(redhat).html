bash
主节点：hostnamectl set-hostname master
工作节点：hostnamectl set-hostname node01

# 修改hosts
cat <<EOF >> /etc/hosts
10.11.8.0 master1
10.11.8.240 master2
10.11.12.126 master3
10.11.9.0 worker1
EOF


# 节点之间免密配置，各个节点配置均可，我这里是master1想其他节点配置免密

##############################################################################
# 参数说明：
以下是这些配置项的含义：

- `net.bridge.bridge-nf-call-iptables`：此参数（设为1）表示在二层转发时，bridge设备也会调用iptables配置的三层规则（包含conntrack）。这样可以解决在Kubernetes环境中，Service同节点通信问题，这也是为什么在Kubernetes环境中，大多都要求开启bridge-nf-call-iptables的原因。

- `net.bridge.bridge-nf-call-ip6tables`：此参数与`net.bridge.bridge-nf-call-iptables`类似，但适用于IPv6。

- `net.ipv4.ip_forward`：此参数用于控制Linux系统的IP转发功能。当值为0时，表示禁止进行IP转发；如果是1，则说明IP转发功能已经打开。

- `net.ipv4.tcp_tw_recycle`：此参数启用了快速回收（Fast Recovery）机制，用于加速TIME_WAIT状态连接的回收。当该参数值为1时，启用快速回收；为0时，禁用快速回收。

- `vm.swappiness`：此参数定义了系统对swap的使用倾向，默认值为60，值越大表示越倾向于使用swap。可以设为0，这样做并不会禁止对swap的使用，只是最大限度地降低了使用swap的可能性。

- `vm.overcommit_memory`：此参数表示内核在分配内存时的检查方式。当值为0时，表示内核将检查是否有足够的可用内存供应用进程使用；如果有足够的可用内存，内存申请允许；否则，内存申请失败，并把错误返回给应用进程。当值为1时，表示内核允许分配所有的物理内存，而不管当前的内存状态如何。当值为2时，表示内核将检查是否有足够的内存（包括RAM和swap）供应用进程使用。

- `fs.inotify.max_user_instances`：此参数表示每一个real user ID可创建的inotify instances的数量上限，默认128。

- `fs.inotify.max_user_watches`：此参数表示每个用户同时可以添加的watch数目，当出现too many open files问题而上面三种方法都无法解决时，可以尝试通过修改这2个内核参数来生效。

- `fs.file-max`：此参数表示Linux内核将分配的最大文件句柄数。当你收到大量关于耗尽文件句柄的错误消息时，你可能需要提高这个限制。

- `vm.panic_on_oom`：此参数用来控制当内存不足时该如何做。当值为0时，内存不足时，启动OOM killer。当值为1时，内存不足时，有可能会触发kernel panic（系统重启），也有可能启动OOM killer。当值为2时，内存不足时，表示强制触发kernel panic，内核崩溃GG（系统重启）。

6、高版本集群cri-docker安装（1.24及以上版本） ​

说明：1.24版本k8s集群弃用dockershim插件，将默认容器运行时改为containerd，因此1.24及1.24以上版本集群如果想使用docker则需要手动安装一个cri-docker

**cri-docker：**cri-dockerd是一个适配器，它提供了一个垫片（shim）以便你可以通过Kubernetes的容器运行时接口（CRI）来控制Docker。Mirantis和Docker已经达成协议，将在Kubernetes之外独立维护shim代码，作为Docker Engine API的符合CRI的接口1。对于Mirantis的客户来说，这意味着Docker引擎的商业支持版本，Mirantis Container Runtime (MCR)，将符合CRI。

bash
# 下载安装
wget https://github.com/Mirantis/cri-dockerd/releases/download/v0.3.1/cri-dockerd-0.3.8-3.el7.x86_64.rpm
rpm -ivh cri-dockerd-0.3.8-3.el7.x86_64.rpm

# 修改cri-docker配置文件，否则kubelet无法拉取镜像
vim /usr/lib/systemd/system/cri-docker.service
# 1.27.6版本的k8s使用 pause:3.9 
ExecStart=/usr/bin/cri-dockerd --container-runtime-endpoint fd:// --pod-infra-container-image=registry.aliyuncs.com/google_containers/pause:3.6

#启动cri-dockerd
systemctl daemon-reload 
systemctl enable cri-docker && systemctl start cri-docker
7、初始化集群 ​
（1）使用命令 ​
bash
kubeadm init \
--image-repository registry.aliyuncs.com/google_containers \
--apiserver-advertise-address=190.5.0.11 \
--kubernetes-version=v1.23.16 \
--pod-network-cidr=10.244.0.0/16 \
--service-cidr=10.96.0.0/12

# 1.24及以上版本需要添加
 --cri-socket=unix:///var/run/cri-dockerd.sock 


##说明：
–apiserver-advertise-address：集群通告地址
–image-repository：由于默认拉取镜像地址k8s.gcr.io国内无法访问，这里指定阿里云镜像仓库地址(此仓库arm64和X86_64均可)，如果不指定默认是k8s.gcr.io
–kubernetes-version：K8s版本，与上面安装的一致
–service-cidr：集群内部虚拟网络，Pod统一访问入口
–pod-network-cidr：Pod网络，与下面部署的CNI网络组件yaml中保持一致
–cri-socket：指定cri-dockerd接口，如果是containerd则使用unix:///run/containerd/containerd.sock

–image-repository registry.aliyuncs.com/google_containers 指定镜像仓库,如果不指定默认是k8s.gcr.io,国内需要翻墙才能下载镜像

（2）使用 ​
bash
#helm基础操作
# 查看仓库
helm repo list

helm repo remove stable
# 添加新的仓库地址
helm repo add stable https://kubernetes.oss-cn-hangzhou.aliyuncs.com/charts
# 更新仓库
helm repo update

#查找 Charts（helm search）
helm search hub 从 Artifact Hub 中查找并列出 helm charts。 Artifact Hub中存放了大量不同的仓库
helm search repo 从你添加（使用 helm repo add）到本地 helm 客户端中的仓库中进行查找。该命令基于本地数据进行搜索，无需连接互联网。

# 查看当前部署的所有 release（helm list）
helm list -n [命名空间名称]
helm list -n default

# 自定义安装chart
# 查看 chart 中的可配置选项
$ helm show values bitnami/wordpress
然后，你可以使用 YAML 格式的文件覆盖上述任意配置项，并在安装过程中使用该文件
$ echo '{mariadb.auth.database: user0db, mariadb.auth.username: user0}' > values.yaml
$ helm install -f values.yaml bitnami/wordpress --generate-name



#创建你自己的charts（helm create）
$ helm create deis-workflow
Creating deis-workflow
#./deis-workflow 目录下已经有一个 chart 了。你可以编辑它并创建你自己的模版，在编辑 chart 时，可以通过 helm lint 验证格式是否正确。
# 当准备将 chart 打包分发时，你可以运行 helm package 命令：
$ helm package deis-workflow
deis-workflow-0.1.0.tgz
# 然后这个 chart 就可以很轻松的通过 helm install 命令安装：
$ helm install deis-workflow ./deis-workflow-0.1.0.tgz
# 打包好的 chart 可以上传到 chart 仓库中
helm push chart.tgz repoName    上传chart到chart仓库
